### Root cause for the original 500s

Production DB was set up before EF migrations were introduced — the Courses,
Modules, etc. tables exist but the `__EFMigrationsHistory` table doesn't have the
initial migration recorded. EF's `Database.MigrateAsync()` tries to apply the
migration, hits "There is already an object named 'Courses'", and aborts.
Newer tables (MadCloudTasks) never get created.

Fix: `SchemaBootstrap.EnsureAsync` runs after MigrateAsync and uses `IF NOT EXISTS`
guards to create only the missing tables. Safe to re-run on every startup; becomes
a no-op once prod is re-baselined.
